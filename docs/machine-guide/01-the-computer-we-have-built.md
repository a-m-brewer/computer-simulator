# The Computer We Have Built

At this point the simulator is no longer just a collection of separate parts. It is a complete small computer, with a CPU, RAM, buses, an I/O path, a display, a keyboard, binary program loading, and an assembler that can turn text into the bytes the CPU understands. That does not make it fast or convenient by modern standards. It does mean that the machine has crossed an important line. A program can live outside the simulator as a file, be loaded into simulated memory, run on the simulated CPU, talk to simulated adapters, and produce visible output.

This guide is about that working machine. It is not another tour through every gate and latch from the beginning. The project still builds the computer from the bottom up, and that matters, but the question changes once the parts can run real programs together. Now we can ask what it means for a program to enter the machine, how the assembler and CPU agree about instruction bytes, how the display and keyboard look like ordinary byte transfers to the CPU, and how small assembly routines can become reusable tools.

The center of the machine is still simple. The CPU fetches instructions from RAM, decodes them, moves values through registers and buses, performs arithmetic or logic in the ALU, and chooses the next instruction address. RAM holds both the program bytes and the data bytes. The CPU has four general-purpose registers, named `R0` through `R3`. The machine's ordinary words are 16 bits wide, but a `DATA` instruction carries only an 8-bit immediate value, so larger constants have to be built in pieces. These are small limits, but they are useful limits because they force the computer's behavior to remain visible.

The simulator keeps that behavior visible in its structure. There is no hidden event system that notices when something changed and wakes up the right object. Components expose an `Update()` method, and the larger parts call those methods in the order needed for one simulated tick. A register does not magically know that a bus changed. It only sees the wires it was given, and it changes when its update step tells it to. This makes the code slower than ordinary software, but it also keeps the simulated machine close to the hardware idea it is meant to model.

One way to picture the whole machine is to follow a very small display program from outside the simulator to the screen. First there is a file of bytes. The file may have been produced by the assembler from an `.asm` source file, or it may be one of the checked-in demo binaries. The simulator does not care where those bytes came from. It reads the file as a raw image and copies the first byte into RAM address `0`, the second byte into RAM address `1`, and so on.

When the computer starts running, the instruction address register begins at the start of memory. During the fetch part of the CPU cycle, the current address is placed where RAM can see it, RAM presents the byte stored there, and the instruction register receives that byte. If the byte is the first half of a two-byte instruction such as `DATA` or `JMP`, the next byte in RAM is also meaningful as an operand. If the byte is a one-byte instruction such as an ALU operation or an I/O transfer, the bits inside that one byte already contain the needed register and mode information.

Suppose the program wants to draw a pattern on the display. The CPU does not draw pixels directly. It first loads the display adapter's I/O address, `0x07`, into a register and sends that value with an address output operation. That selects the display adapter. Then it sends another address, this time meaning a byte location inside display RAM. Finally it sends one data byte. The display adapter stores that byte. One display RAM byte represents eight horizontal pixels, so the byte is not just a number anymore. It has become a short row of light and dark dots.

Nothing about that sequence requires the CPU to understand a terminal window. The CPU moved bytes. The display adapter interpreted those bytes as display memory. Later, the display renderer asks the adapter for a frame and turns display RAM into something the host terminal can show. The renderer has practical choices, such as scanning faithfully through the simulated display logic or using a faster buffer path, but those choices sit outside the CPU's instruction set. The program still talks to the display by selecting an adapter and writing bytes.

The keyboard follows the same pattern in the other direction. A host terminal key press is queued for the keyboard adapter. A program selects the keyboard's I/O address, `0x0F`, and then performs a data input operation. If a key is waiting, the adapter provides its ASCII value. If no key is waiting, it provides `0`. A program that echoes text on the screen is therefore a loop made from small pieces: ask the keyboard for a byte, ignore zero, find the glyph for a non-zero character, and write that glyph into display RAM one row at a time.

This is why the machine feels larger than its instruction set. The CPU still has a small number of operations. It can load and store, add and shift, compare and jump, and move bytes through the I/O bus. The useful behavior comes from arranging those operations carefully. A display pattern, a `HELLO WORLD` message, and an interactive echo program are all made from the same small actions. The difference is not hidden hardware. The difference is the program.

There are also honest limits. The CPU has no stack, so reusable routines use explicit registers and scratch RAM instead of ordinary call and return instructions. It has no interrupts, so a key press or timer cannot pause the current program and send the CPU to a special handler. Short jumps use one-byte targets, and longer jumps are built as software sequences. The display is currently monochrome. The simulator is intentionally faithful enough to be slow compared with normal software. These limits are not mistakes to hide from the reader. They are part of the shape of this computer, and they explain why later sections spend time on loaders, assemblers, adapters, and conventions.

A stack is usually a simple memory area that grows and shrinks as a program calls routines. It gives each call a place to save a return address, temporary values, and sometimes arguments, so one routine can safely call another and then come back. An interrupt is a different kind of help from the hardware: it lets an outside event, such as input becoming available, make the CPU stop what it was doing, run a small service routine, and then return. This machine does neither of those things automatically. Programs have to keep their own return addresses and scratch values in known places, and they have to ask the keyboard whether a key is waiting instead of being interrupted by one.

The next question is how all of these pieces agree about a program. The CPU only sees bytes, while a person wants to write names such as `ADD`, `JMP`, and `OUT`. If the assembler and the CPU do not speak exactly the same byte language, the whole machine becomes unreliable. So before we put more programs into the computer, we need to look at where that language is defined.
