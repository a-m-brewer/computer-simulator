Programs Written in Their Own Language

Once an assembler exists, there is an important question to ask about the simulator's own demonstration programs. Should they be built by C# code that calls instruction helpers, or should they be written in the same assembly language that a person would use? The first approach can work, but it keeps the examples one step away from the tool we just built. The second approach is more demanding and more useful. It asks the assembler to produce the real programs that the simulator runs.

This is often called dogfooding: using the tool for the work it is meant to do. In this repository, the main demonstration programs live as `.asm` files under [`programs/`](../../programs/). They are assembled into raw `.bin` images. The simulator then loads those binary images exactly as it would load any other program. The runtime path does not become special just because the program is a built-in demo. The source is assembly, the artifact is binary, and the computer still executes bytes from RAM.

The display pattern program is a good first example because its behavior is simple and visible. Its source includes the common I/O constants, loads the number of display bytes in the frame, selects the display adapter, and then enters a loop. In that loop, the current display RAM address is sent with `OUT ADDR`, the same value is sent as the pixel byte with `OUT DATA`, the address is incremented, and the program compares the new address with the frame size. Until the end is reached, it jumps back to the loop. The visible result is a display filled with bytes whose bit patterns come from their own addresses.

There are several layers hiding behind that short description. The source name `IO_DISPLAY` comes from an included assembly file, not from the CPU. The name `BYTES_PER_FRAME` is supplied at assembly time, because it depends on the display dimensions being targeted. The label `Loop` is resolved by the assembler into a numeric jump address. `HALT` becomes a self-jump. After assembly, none of those source conveniences remain. The checked binary is only the emitted instruction and operand bytes, ready for [`ProgramLoader`](../../ComputerSimulator.Core/ProgramLoader.cs) to copy into RAM.

The text programs show why this matters even more. [`hello-world.asm`](../../programs/hello-world.asm) is not just a handful of instruction bytes. It includes display routines, places a message string in the image, and uses `.incbin` to place the shared 8x8 font image at the agreed font base address. When the program runs, it asks the display routine to print the zero-terminated string. The routine reads the characters, finds each glyph in the font data, and writes eight rows of pixels for each character to display RAM. The simulator is not secretly drawing the words. The assembled program is doing the work with memory, registers, I/O instructions, and the display adapter.

The [`echo`](../../programs/echo.asm) program goes one step further because it is interactive. Its assembly source selects the keyboard, polls for a byte, ignores zero when no key is available, handles Enter and Backspace, and writes printable characters to the display by copying glyph rows. It is long enough that writing it as raw bytes would be miserable, and it is still close enough to the machine that every part of the behavior is visible. Polling, comparing, branching, cursor storage, font lookup, and display writes all appear as ordinary assembly operations.

Following one program from source to screen makes the relationship clear. Start with [`display-pattern.asm`](../../programs/display-pattern.asm). The assembler loads the source file, expands the include for I/O constants, applies the supplied value for `BYTES_PER_FRAME`, records the loop label, expands any pseudo-instructions, and emits the bytes. For the default display, a checked-in binary under [`programs/bin/`](../../programs/bin/) holds the result. When the simulator starts that built-in demo, it finds the binary path, reads the bytes, loads them into RAM beginning at address `0`, and starts updating the computer. The CPU fetches the first instruction, eventually selects the display adapter, and writes display RAM one byte at a time. The renderer later presents those bytes as pixels.

The [`hello-world.asm`](../../programs/hello-world.asm) path has the same shape but a richer image. The source begins by jumping over included routines so execution starts at the program's own entry point. It places the [display standard-library code](../../programs/stdlib/display.asm) near the beginning of memory, places the message text after the program logic, and places the [font binary](../../programs/assets/font8x8.bin) at `FONT_BASE`. When assembled, all of that becomes one continuous binary image with zero-filled gaps where `.org` moved the output address forward. Loading the image into RAM gives the program both its instructions and its data.

When that hello-world program runs, the CPU does not know that one region of RAM came from an include file and another came from a string directive. It only follows instructions. The program puts the message address in `R0`, puts the display byte address in `R2`, puts a return address in `R3`, and jumps to the print routine. The routine reads one character byte, finds the eight font bytes for that character, writes them to display RAM rows separated by `BYTES_PER_ROW`, and repeats until it reaches the zero terminator. This is a useful demonstration because the source language, assembler directives, memory layout, and display adapter all meet in one visible result.

The echo program uses the same machinery in a less predictable setting. Its source cannot know which key the user will press next, so it assembles the loop that will make that decision at runtime. The program selects the keyboard address, reads data, compares the result with zero, and branches back if no key is available. For a printable character, it calculates the glyph address and writes pixels. For Enter or Backspace, it changes cursor state stored in RAM. The assembler prepares the code, but the running CPU still makes the decisions one byte at a time.

The checked-in binaries are useful because built-in demos should run without requiring the user to assemble them first. They also create a risk: the source and binary can drift apart. If someone changes `hello-world.asm` but forgets to update `hello-world.bin`, the demo might no longer represent the source file beside it. The integration tests guard against this by assembling the source fresh and comparing it with the checked-in binary for the default display configuration. When the bytes differ, the test points out that the generated artifact is stale.

Those tests are more than file checks. The display pattern test assembles the source, loads the emitted bytes into a simulated computer, runs the machine long enough for the program to write display RAM, renders a frame, and checks that each pixel matches the address-derived pattern. That path is intentionally long because it exercises the same route a real demo uses: source, assembler, binary image, program loader, CPU execution, adapter writes, and display output. A shorter test could prove fewer pieces.

This arrangement keeps the simulator honest in two directions. The assembler has to be good enough to build the programs people actually run. The runtime has to remain simple enough to run raw binaries without learning assembly syntax. If a feature works only when a C# test constructs instruction bytes directly, it is not yet proven as a language feature. If a demo works only because the simulator recognizes a special program name and bypasses normal loading, it is not yet proven as a machine program. The dogfood path ties those pieces together.

It also makes the examples better teaching material. A reader can open the assembly source and see the same operations described in these sections: selecting adapters, writing display RAM, reading the keyboard, copying font rows, using labels, using pseudo-instructions, and placing binary assets. The examples are not translations from some hidden source. They are the source. The `.bin` files are merely the form the simulated computer needs in order to run them.

There is still a boundary to respect. The assembly programs are not magically portable across every display size unless the needed constants are supplied at assembly time. The default binaries are built for the default display. A different width changes values such as bytes per row or bytes per frame, so a custom run should assemble the source with the matching `-D` values and then run the generated binary. The machine stays simple, and the assembly step is where those outside choices are turned into numbers.

At this stage, the simulator has a useful loop of its own. The instruction set defines the bytes. The assembler emits those bytes from readable source. The dogfood programs use the readable source for real demos. The loader places the emitted bytes in RAM. The CPU and peripherals produce visible behavior. That loop is what turns an assembler from a separate convenience into part of the way the computer is used.
