Words That Save Work

Assembly language removes some of the burden of writing raw bytes, but it can still be repetitive. The CPU has a small instruction set, and that is a strength of the simulated hardware. The cost is that a simple thought in a programmer's head may take several machine instructions to express. A 16-bit address cannot fit in the one-byte immediate value used by `DATA`. Copying one register to another is not a single hardware instruction. Stopping a program is really a jump that returns to itself. These are not flaws in the assembler. They are places where a tool can save work while leaving the machine unchanged.

A pseudo-instruction is one of these saved pieces of work. It looks like an instruction in the source file, but the CPU does not have a matching instruction byte for it. The assembler expands it into real instructions that the CPU already understands. This distinction is important. If a program contains `LDI`, there is no new `LDI` circuit in the CPU. There is only assembler logic that decides which real bytes to emit. Once the `.bin` file exists, the pseudo-instruction has vanished.

The simplest example is loading a small value. The CPU already has `DATA`, which places the next byte from the instruction stream into a register. If the programmer writes `LDI R0, 42`, the value fits in one byte, so the assembler can emit exactly the same bytes it would have emitted for `DATA R0, 42`. In this case the pseudo-instruction is mostly a clearer word. It says load an immediate value, and the assembler chooses the shortest real form.

The reason `LDI` becomes more useful is that registers are 16 bits wide while `DATA` carries only an 8-bit immediate operand. A program often needs an address such as `0x2000`, especially when it is pointing to font data or a routine placed later in memory. The CPU cannot fetch that whole word as one immediate value. The assembler can still help by building the word out of the smaller operations the CPU already has.

For a word-sized `LDI`, the programmer must name a scratch register. If the destination is `R0`, the value is `0x1234`, and the scratch register is `R3`, the assembler emits a sequence with a clear shape. First it loads the low byte, `0x34`, into `R0`. Then it loads the high byte, `0x12`, into `R3`. Then it shifts `R3` left eight times, moving that high byte into the upper half of the 16-bit word. Finally it adds `R3` into `R0`. The CPU only performs `DATA`, `SHL`, and `ADD`, but after those instructions run, `R0` contains the word the programmer asked for.

That expansion also explains why the scratch register is explicit. The assembler cannot silently borrow a register without risking damage to a value the program still needs. On a four-register machine, every register matters. By writing the scratch register in the source, the programmer makes the clobber visible. The assembler does not pretend that the temporary work is free; it merely writes the repeated instructions correctly.

`JMP16` solves a related problem. The ordinary `JMP` instruction uses a one-byte target, so it can only jump to addresses that fit in that byte. This is enough for short programs and local loops near the beginning of memory, but it is not enough for a source file that places data or routines beyond `0x00FF`. The hardware already has `JMPR`, which jumps to the address held in a register. `JMP16` combines these pieces. The assembler loads the full target address into a named register, using a named scratch register if needed, and then emits `JMPR` for the address register.

This is still a software long jump. It does not make the CPU's short jump wider. If a program uses ordinary `JMP` or a conditional jump, the target must still fit in one byte. If the target is farther away, the source must use the longer software form and provide the registers needed to build the address. That limitation is useful to keep visible, because it reminds us where the real machine stops and where the assembler starts helping.

Other pseudo-instructions follow the same rule. `MOV R1, R0` can be expanded by clearing `R1` with a data load of zero and then adding `R0` into `R1`. `HALT` can be expanded into a jump to its own address, because a CPU with no separate halt wire can be made to stay in one place by repeatedly fetching the same jump. When the halt address no longer fits in a short jump, the longer halt form uses registers in the same spirit as `JMP16`.

These words make source programs easier to read, but they also make program size less obvious. A small `LDI` is two emitted bytes. A word-sized `LDI` is much larger because it includes two data loads, eight shifts, and an add. `JMP16` includes the cost of loading the address before the register jump. This is why the assembler performs layout before final emission. A pseudo-instruction may be compact or expanded depending on the value it has to represent, and labels after it must receive the address that matches the actual emitted bytes.

That size difference is not only an accounting detail. Imagine a label placed immediately after two `LDI` lines. If both values fit in one byte, the label will be close. If one value later becomes a full 16-bit address, the expanded instruction sequence pushes the label farther away. A hand-written address would now be wrong, but a label can be recalculated. The assembler counts the real emitted bytes so the final jump targets and data addresses match the expanded program, not the shorter source text.

This also explains why pseudo-instructions and labels work well together. A source file can say `JMP16 Message, R0, R3` before the final address of `Message` is known. The assembler first estimates the layout, learns where `Message` lands, and then emits the load-and-jump sequence using that address. If the message moves because another string or routine was inserted before it, the source line does not have to change. The generated bytes change because the address behind the name changed.

The assembler's conveniences are most useful when they preserve the small-machine feeling. `LDI` does not let a register receive a value from nowhere; it builds the value using immediate bytes, shifts, and addition. `JMP16` does not let the instruction stream carry a 16-bit jump target; it loads a register and uses the real register jump. `MOV` does not create a new copy circuit; it clears and adds. The expanded form may be longer, but it remains explainable as ordinary CPU work.

There is a tradeoff for the programmer. Pseudo-instructions make source easier to write, but the programmer still needs to remember the effects that matter. A large `LDI` clobbers the scratch register. A long jump uses the address register. A short jump still has a one-byte limit. A halt is not a power switch; it is a loop that keeps the instruction address from moving forward. Good assembly source is allowed to be convenient, but it should not let the reader forget these costs.

There is a useful discipline here. Pseudo-instructions should feel like shorthand for work the programmer could have written by hand. They should not hide a new device, a new register, or a new CPU behavior. The assembler may choose bytes, fill addresses, include files, and repeat simple instruction patterns. The simulator still runs only the instruction set it already has. That line keeps the source language friendly without letting the tool blur what the computer can really do.

With pseudo-instructions, assembly becomes less like counting bytes and more like giving compact names to common machine chores. The program can say load this address, move this value, or jump to this far label, while the assembler writes the necessary small steps. The next step is to use this language not just in tests or examples, but for the programs that the simulator itself demonstrates.
