# The Same Language in One Place

A computer program for this machine is a row of bytes. A person may write `ADD R3, R0` in an assembly file, but the CPU never sees those words. It sees one byte whose bits say which operation to perform and which registers are involved. That is fine as long as every part of the project agrees about the meaning of the byte. If the CPU thinks one bit pattern means one thing, and the assembler emits that same bit pattern for something else, the simulator can still run, but it will run the wrong program.

This is why the instruction language has to live in one shared place. The simulator has several different readers of the same machine language. The CPU control logic decodes instruction bytes while the machine runs. The assembler turns text into instruction bytes before the machine runs. Tests build small programs and assert exact encodings. Demo binaries are checked against source programs. If each of those places kept its own private copy of the instruction numbers, the copies would eventually drift apart. A one-byte mistake is enough to turn a branch into the wrong branch, or an addition into an addition that stores its answer in the wrong register.

The shared definition is the instruction set. In this repository, [`InstructionSet`](../../ComputerSimulator.Core/Instructions/InstructionSet.cs) is not a second CPU. It is a compact encoder for the byte language the CPU already understands. It knows that a load instruction uses one prefix, a store instruction uses another, an I/O instruction uses the `0111` prefix, and ALU operations use their own operation bits. The assembler calls this shared encoder when it emits bytes. Tests call it when they need to express an instruction without writing a raw number by hand. The result is that ordinary code can talk about instructions by name while the binary image still contains plain bytes.

Consider `ADD R3, R0`. In assembly, the mutating operations are written destination first, so this means `R3 = R3 + R0`. That is the order many programmers expect to read. The hardware encoding, however, names two register fields in the instruction byte, and the destination is the second register field. The assembler has to bridge those two views. It reads the destination register `R3`, reads the source register `R0`, and asks the shared instruction set to encode an add with source `R0` and destination `R3`. The emitted byte is the machine instruction, not the text.

The important part is not the particular number, although the number is real. With this encoding, `ADD R3, R0` becomes `0x83`. The important part is that the assembler did not invent that number on its own. It used the same instruction table that the tests use when they describe expected programs. If someone accidentally encoded the registers in written order instead, the byte would describe a different operation to the CPU. The program might add the same two values, but it would place the result in `R0` instead of `R3`, and the next instruction would be working with the wrong state.

Two-byte instructions show the same idea in a slightly different way. `DATA R1, 42` is not a 16-bit instruction. It is an instruction byte followed by an operand byte. The first byte says that the CPU should load immediate data into `R1`. The next byte is the value `42`. When the program is in RAM, those are just neighboring bytes. The CPU's sequencing gives them meaning by fetching the instruction byte, recognizing that it needs an operand, and taking the following RAM byte as data instead of as a separate instruction.

Jumps are another place where shared encoding matters. The unconditional short jump has its own instruction byte, followed by a one-byte target address. Conditional jumps use flag selections. The ALU records conditions such as carry, above, equal, and zero in the CAEZ flags, and a conditional jump byte says which of those flags should allow the branch. If the assembler says `JE Done`, it must emit the byte for the equal flag and then the one-byte address of `Done`. A wrong flag bit might still look like a valid instruction, but it would test the wrong condition.

I/O instructions also depend on this common language. `OUT ADDR, R1` and `OUT DATA, R1` are both I/O instructions using the same prefix. One bit says output rather than input. Another bit says address rather than data. The low bits name the register. The CPU uses those bits to drive the I/O control wires, and the display or keyboard adapter responds to those wires. If the assembler duplicated the I/O constants and got one of those bits wrong, a program might try to select the display but actually send data, or try to read from the keyboard but drive the bus in the wrong direction.

This does not mean every convenience in assembly is a hardware instruction. Some words in the assembly language are pseudo-instructions. `MOV` makes copying registers easier, but it expands into real instructions the CPU already has. `LDI` may be one `DATA` instruction for a small value, or a longer sequence for a 16-bit value. `HALT` is implemented as a jump back to itself. These conveniences still pass through the shared instruction set when the assembler emits their real bytes. They make programs easier to write without adding secret abilities to the CPU.

There is a useful discipline here. Raw byte constants are still appropriate in a few tests that are specifically checking the binary contract, but normal program construction should not scatter magic instruction numbers through the repository. A name such as `InstructionSet.JumpIf(JumpCondition.Equal)` says what is being built and leaves the exact bit pattern to the one place that owns it. That makes mistakes easier to find and future changes easier to reason about.

The CPU and the assembler are separate tools, and they should stay separate. The simulator does not parse assembly source. The assembler does not run the simulated machine. They meet at the byte image. The shared instruction set is the dictionary that keeps that meeting honest: the assembler uses it to write the bytes, and the simulator's CPU is wired to understand those bytes when they are fetched from RAM.

Once the language is centralized, the next problem is physical in the simulated sense. A program can now be expressed as the right bytes, but those bytes still have to enter RAM. That is the job of the loader.
