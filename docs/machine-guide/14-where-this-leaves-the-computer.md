Where This Leaves the Computer

The machine now has more than a CPU and a memory. It has a path from a program a person can write to a visible result a person can see. The CPU still fetches one instruction byte at a time, decodes it, changes registers, reads or writes RAM, and chooses the next instruction address. Around that center there is now an [I/O bus](../../ComputerSimulator.Core/Parts/IoBus.cs), a [display adapter](../../ComputerSimulator.Core/Peripherals/Display/DisplayAdapter.cs), a [keyboard adapter](../../ComputerSimulator.Core/Peripherals/Keyboard/KeyboardAdapter.cs), a [loader](../../ComputerSimulator.Core/ProgramLoader.cs) for binary images, an [assembler](../../ComputerSimulator.Assembler/) that translates source text into those images, dogfood programs written in the machine's own assembly language, and reusable routines that keep common work from being rewritten in every program.

That is an important change in how the computer feels. Earlier, a program could be described as a list of bytes placed directly into RAM. That is still what the CPU finally sees, but it is no longer what the programmer has to think about first. The programmer can write a label, a mnemonic, a register name, or a string of characters. The assembler can turn those choices into bytes. The simulator can load the bytes. The adapters can make some of those bytes mean pixels and key presses. Each layer is small, but the layers together make the machine approachable.

Consider the whole trip for one character appearing on the screen. A program begins as assembly source. Somewhere in that source is a message byte, perhaps part of a zero-terminated string. Somewhere else is a call to a display routine, made by putting the message address in `R0`, the display byte address in `R2`, the return address in `R3`, and jumping to the print routine. The source also includes the display library and places the font image in RAM, because a character cannot be drawn until its eight rows of pixels are available to the program.

When the assembler runs, it resolves the labels in that source. The label for the message becomes a numeric RAM address. The label for the return point becomes another address. The included routine source becomes part of the same output image. Pseudo-instructions such as loading a full word or making a long jump are expanded into real instruction bytes. The result is not a special assembly object that the simulator understands. It is a raw binary image, a sequence of bytes that can be copied into RAM starting at address zero.

When the simulator runs that image, the first useful thing it does is much less literary. The loader writes byte zero of the image to RAM address zero, byte one to RAM address one, and so on. The CPU begins fetching from the start. It may immediately jump over library routines to the program's entry point, because routine bodies are data to be executed later, not the first thing the program wants to do. From there the CPU executes the instructions the assembler emitted. It fills registers, reads message bytes from RAM, jumps into the print routine, and returns when the routine is finished.

The display adapter enters the story only when the program uses the I/O bus. The routine selects the display's I/O address, sends a display RAM byte address, and sends a pixel byte. It repeats that for the eight rows of the glyph. If the character is `A`, the value in RAM is still just the ASCII number for `A` until the routine uses that number to find the glyph rows in the font. Each glyph row is still just a byte. Bit 0 is the leftmost pixel in that display byte, and the next row of the same character is `BYTES_PER_ROW` display bytes later. After the writes have happened, display RAM contains a small picture rather than a character code.

Finally the host side renders the display. The faithful version can scan through simulated display behavior, and the faster path can walk display RAM bytes directly when producing a terminal frame. Either way, the result visible to the person is not produced by a secret text mode. It is produced by the same chain of byte-level choices the program made: source text became instruction bytes, instruction bytes moved register values, register values selected an adapter, adapter writes changed display RAM, and display RAM became pixels.

The keyboard follows the same pattern in the other direction. A real key press arrives through the host terminal and is queued as a byte. The simulated program does not receive an event. It selects the keyboard adapter and performs an `IN DATA` read. If a key is waiting, the adapter presents the ASCII code. If not, it presents zero. An echo program can poll for a non-zero byte, decide whether the byte is Enter or Backspace, and then use the same display routines to show or erase a glyph. The interaction looks immediate to a person, but inside the simulator it is still built from repeated updates and explicit bus transfers.

This leaves the computer in a good middle place. It is still simple enough that a curious reader can trace a value from one part to the next. There is no hidden operating system, no hardware text console, no stack, no color display, and no disk. The machine is also no longer a bare collection of parts. It can load programs from files. It can assemble readable source into binary. It can draw text from a font, accept keyboard input, and reuse assembly routines for printing, reading, copying, multiplying, and dividing.

The remaining limitations are not failures of the design. They are useful markers of what has and has not been built. A stack would make nested calls and local storage much easier. Wider or richer display behavior would make the screen more expressive. More devices would give programs more places to send and receive bytes. Faster rendering paths already show one practical compromise between faithful simulation and usable output. Each possible improvement can be understood against the same question that guided the work so far: can the new behavior be explained as wires, components, instructions, adapters, or ordinary programs running on the machine?

That question is what keeps the project coherent. The simulator began with tiny logical parts and grew upward. The later conveniences did not erase that foundation. The assembler does not teach the CPU assembly language; it emits bytes the CPU already knows. The standard library does not add procedure hardware; it follows a calling convention in RAM and registers. The display does not know about strings; it stores pixel bytes. The keyboard does not run the program; it offers one byte when selected and read. The computer is more useful now because each new layer respects the smaller layers beneath it.

This also gives future work a clear test. If a new feature makes the machine easier to use, it should still be possible to say where it belongs. Some work belongs in assembly, as a routine or a program. Some work belongs in an adapter, because it is a new kind of device on the I/O bus. Some work belongs in the host, because it is about showing the simulated state to a real terminal. Keeping those boundaries clear lets the simulator grow without turning the CPU into a place where every convenience is hidden.

This is a useful place for the guide to leave the machine: not finished in every possible direction, but complete enough to be used, inspected, and extended. A person can write a small assembly program, assemble it, load it, run the simulator, type on the keyboard, and see pixels change on the display. More importantly, that person can still explain how it happened without skipping over magic. The path is long, but every step is made from the same modest operations the computer has been using all along.
