A Screen That Can Keep Up

The [display adapter](../../ComputerSimulator.Core/Peripherals/Display/DisplayAdapter.cs) can now remember pixel bytes, but remembering pixels and showing pixels are not the same job. The CPU writes [display RAM](../../ComputerSimulator.Core/Peripherals/Display/DisplayRam.cs) when a program asks it to. The person using the computer only sees a picture after the simulator turns that display RAM into terminal output. That second job can become surprisingly expensive, because this project tries to model the computer from small parts upward. A faithful screen scan is not just a loop that copies an array. It uses simulated counters, simulated address registers, display RAM reads, and one pixel decision after another.

The slow version is valuable because it shows what the display hardware is doing. A screen is scanned from left to right and from top to bottom. At the beginning of each eight-pixel group, the display logic calculates which display RAM byte belongs to the current position, loads that address into the read side of display RAM, and reads the byte. Then, for each pixel in that group, it chooses one bit from the byte. If the current horizontal position is at bit 0, it uses bit 0. If the current horizontal position is at bit 7, it uses bit 7. After each pixel, the horizontal position advances. At the end of a row, the horizontal position wraps to zero and the vertical position advances.

That is an excellent model of a raster scan, but it is a lot of simulated activity for a host terminal. On the default 96 by 48 display, there are 4,608 pixels in one complete frame. A gate-level scan visits every one of them, even when the program has changed only one display RAM byte since the last frame. A larger display multiplies the cost quickly. A 320 by 200 display has 64,000 pixels. If each of those pixels asks several simulated components to update, the renderer can spend most of its time proving again that blank or unchanged pixels are still blank or unchanged.

The important distinction is between the machine's [scan mode](../../ComputerSimulator.Core/Peripherals/Display/DisplayScanMode.cs) and the host's renderer. The scan mode is how the display adapter decides what pixels should be sent to an output. The renderer is where those pixels go. The output might store pixels in a terminal buffer, write block characters, or compress groups of pixels into braille characters. Changing the terminal's pixel mode changes the shape of the host output, but it does not by itself change whether the display adapter performed a faithful pixel-by-pixel scan or a faster display-RAM walk.

| Question | Gate-level scan | Scan-buffer mode |
| --- | --- | --- |
| What does it walk? | Every pixel position | Display RAM byte addresses |
| What does it skip? | Nothing in a rendered frame | Unchanged bytes after the first frame |
| What should it show? | The current display RAM contents | The same display RAM contents |
| Why keep it? | It matches the hardware model closely | It keeps interactive display refresh practical |

The gate-level scan starts at the top-left pixel. It asks the [screen-control logic](../../ComputerSimulator.Core/Peripherals/Display/ScreenControl.cs) for the current `x` and `y`. When `x` is on a byte boundary, the screen control addresses the current display RAM byte and display RAM exposes that byte on its output bus. Then the screen control sets a brightness wire from the bit that matches the low three bits of `x`. The adapter sends that brightness value to the display output and moves to the next pixel. This repeats until the whole frame has been presented. Nothing about this path assumes that the display output is a terminal. It is a scan of simulated display memory.

The faster scan-buffer mode uses a different observation. Display RAM already knows when a byte has changed, because the write side can compare the stored value before and after a write. If no display byte has changed since the last frame, there is no new picture to present. If only one byte changed, only eight pixels need to be refreshed in the host output. The adapter can walk display RAM byte addresses directly, turn each changed byte into eight pixels, and then present the completed frame. On the first frame it still scans all display bytes so the output starts from a complete picture. After that, it can skip unchanged frames and update only changed byte positions.

There is still a frame boundary in both modes. The display adapter may call `SetPixel` many times, or it may use the byte-oriented output path when the output supports it, but the person should see a completed frame rather than half of a change. That is why the output has a `Present` step. The adapter first pushes the pixels or pixel bytes that belong to the frame. Then it asks the output to present them. The [terminal renderer](../../ComputerSimulator/Tui/TerminalFrameRenderer.cs) can use that separation to update an off-screen buffer and flush the finished picture, which avoids treating every individual pixel write as something the user must immediately see.

The dirty-byte set is deliberately a display concern, not a new CPU feature. The CPU does not ask which pixels changed. It does not receive an interrupt when the display is ready. It just keeps running instructions and writing bytes. Display RAM records changed addresses because the render path can use that information later. This keeps the machine model clean: writes still happen through the I/O adapter, while rendering policy stays on the display side of the adapter.

Consider the smallest useful comparison. A program writes `00000001` to display RAM address `0`, and no other display bytes change. In gate-level mode, rendering a frame still means sweeping the whole screen. The scan will eventually reach pixel `(0, 0)`, see bit 0 of byte 0, and mark that pixel as on. It will also visit every other pixel and mark each one according to its byte. In scan-buffer mode, the display RAM records that address `0` became dirty when the write happened. Rendering can convert byte address `0` to row `0` and byte column `0`, send that one byte to the output, clear the dirty set, and present. The visible result is the same single lit pixel, but the path to produce it is much shorter.

Now consider a program that fills the whole screen one byte at a time. During the fill, the CPU is still the slow part. Each display byte is written through normal instructions, through the CPU bus, through the I/O bus, and into display RAM. Scan-buffer mode does not make those CPU writes happen faster. It only makes the RAM-to-terminal refresh cheaper once some bytes have changed. This matters when looking at performance numbers. If the program is still spending most of its time executing simulated instructions, changing the display scan mode will not make the program draw into display RAM at the speed of ordinary host software.

The terminal output has its own choices after it receives pixels. In block mode, one terminal character represents one simulated pixel, so the picture is easy to reason about but takes a lot of terminal space. In braille mode, one terminal character can represent a small group of pixels, so a larger simulated display fits into a smaller host window. Both modes are render-target choices. They are useful for viewing the machine, but the display memory format remains the same: one byte is eight horizontal pixels, and bit 0 is the leftmost pixel in that byte.

This is why the simulator keeps both paths. The gate-level scan is the clearest continuation of the hardware model. It shows that a screen can be built from counters, address selection, memory reads, and a brightness wire. The scan-buffer path is the practical way to run interactive demos without spending unnecessary host time on unchanged pixels. Both paths must agree on what the screen looks like. They only differ in how much simulated work they perform to arrive at that picture.
