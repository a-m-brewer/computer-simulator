The Language Becomes a Tool

Up to this point, the computer has been able to run programs only after those programs have been turned into bytes. That is the form the CPU understands. A byte in RAM might mean load a value into a register, jump to another address, write to the I/O bus, or perform an ALU operation. The CPU does not see names like `Loop` or `R2` as names. It sees bit patterns. This is honest to the hardware, but it is a difficult way for a person to write anything longer than a few instructions, because the person has to remember every instruction number, every operand bit, and every address that might move when the program changes.

The assembler solves that problem without changing the computer. It is a tool outside the simulated machine. It reads a text file written in assembly language and produces a raw `.bin` file made of bytes. The simulator still does not parse assembly. It still receives bytes through [`ProgramLoader`](../../ComputerSimulator.Core/ProgramLoader.cs), places those bytes into RAM, and lets the CPU fetch them. The new thing is that a person can now describe the program in a form that keeps the intent visible while the assembler performs the bookkeeping.

The most important bookkeeping is turning names into numbers. A label is a name for the address where a later instruction or piece of data will be placed. When a program says that a jump should go to `Loop`, the CPU cannot use the word `Loop`. The assembler must decide what byte address `Loop` means, then put that number into the emitted program image. If an instruction is added before the label, the address changes. The source code can keep saying `Loop`; the assembler recalculates the address the next time the program is assembled.

For a small example, imagine a program that starts with a label named `Loop`, then reads a keyboard byte, compares it with zero, and jumps back to `Loop` if no key was available. The source text uses the label because that is what the programmer means: go back to the polling point. During assembly, the label is recorded as the current output address. Later, when the conditional jump is emitted, the assembler writes the real jump instruction byte and then writes the numeric address that was recorded for `Loop`. In the binary file, the name is gone. Only bytes remain.

This gives assembly language a useful double nature. It is still close to the machine, because each real instruction corresponds to the CPU's existing instruction set. `OUT ADDR, R1` still means an I/O output instruction with the address/data bit set for an address and the register bits set for `R1`. `ADD R3, R0` still becomes one ALU instruction byte. But assembly is also a little more humane than raw binary, because it lets the text say which register is involved, which label is the destination, and which number is being used.

The assembler also has to understand the shape of a program before it can safely emit the final bytes. Some source lines produce one byte. Some produce two bytes. Directives such as `.byte`, `.word`, `.ascii`, `.asciz`, `.org`, `.include`, and `.incbin` may add data, reserve space, or bring in other files. A label does not add a byte at all; it marks the address reached so far. Because later labels can depend on the size of earlier source lines, the assembler lays out the program, resolves symbols, and then emits the final image.

That final image is deliberately plain. It is not an object file with sections, metadata, relocation tables, or a loader script. It is just the bytes that should appear in RAM. If the first emitted byte is the instruction at address `0`, then `ProgramLoader` stores it at RAM address `0`. If the second emitted byte is an operand, it goes at address `1`. If `.org` moves the output address forward, the assembler fills the gap with zero bytes so that the later bytes land at the requested address. The simulator receives the result the same way no matter how pleasant or unpleasant the source file was to write.

The [command-line assembler](../../ComputerSimulator.Assembler.Cli/Program.cs) is only a small wrapper around this translation. It chooses an input file, chooses an output file, and accepts simple definitions with `-D`. Those definitions let the same source program adapt to values that are known outside the program text, such as the number of display bytes per row. The assembler replaces the symbolic name with the provided number while building the output. The CPU never learns that the name existed.

Before the assembler can emit anything, it has to read the text as text. It removes comments that begin with semicolons, recognizes labels by their trailing colon, treats mnemonics and registers without caring about case, and leaves labels case-sensitive so that names do not accidentally merge. A register name such as `R2` becomes a register operand. A number such as `0x2A`, `42`, or `'A'` becomes a value operand. A memory operand such as `[R0]` becomes an address-through-register operand. This parsing step is not yet machine execution; it is only the tool turning a line of text into pieces it can reason about.

After parsing, the assembler has to speak the exact same byte language as the simulator. That is why it uses the [shared instruction definitions](../../ComputerSimulator.Core/Instructions/InstructionSet.cs) from the core project rather than inventing its own copy of the opcode table. If the assembler duplicated the numbers and made a mistake, the source file could look correct while the CPU received the wrong byte. The shared instruction set keeps `ADD`, `JMP`, `OUT`, and the rest of the language tied to the same encodings the CPU already decodes.

Operand order is another place where the assembler acts as a translator rather than a new machine. In source, mutating ALU operations are written destination-first, so `ADD R3, R0` reads like `R3 = R3 + R0`. The hardware encoding still has its own register fields. The assembler maps the source order into the existing encoding and emits the byte that makes the CPU put the result in the intended register. The source language is chosen for the human reader, and the emitted byte is chosen for the hardware that already exists.

This separation matters because it keeps the layers honest. The core simulator defines the instruction bytes. The assembler uses those definitions when it emits a program. The loader loads bytes and does not care where they came from. The CPU fetches and executes bytes and does not care whether a person wrote them by hand, the assembler emitted them from text, or a test built them directly. Each layer has a smaller job because the assembler is a tool around the machine, not new hardware inside it.

A useful way to picture the flow is to follow one jump. In the assembly source, the programmer writes a label before the repeated work and writes a jump to that label after the test. The parser turns those lines into statements. The layout pass counts the bytes that each statement will produce and records the label's address. The emit pass writes the jump instruction and the recorded address into the output. [`ProgramLoader`](../../ComputerSimulator.Core/ProgramLoader.cs) copies those bytes into RAM. When the CPU later reaches the jump, it loads the byte after the jump instruction as the next instruction address. By then, the name has done its job and disappeared.

The same kind of translation happens for data. A directive such as `.asciz` is not an instruction for the CPU to execute. It is a request to place bytes into the program image, ending with a zero byte so that a program can later find the end of the string. A directive such as `.incbin` is even more direct: copy the contents of another binary file into this program image at the current address. These directives make the source file a layout description as well as an instruction list. Once loaded, the CPU sees the result only as bytes in RAM.

One helpful mental model is that assembly has two audiences. The programmer reads the source file, so the source should say `Message`, `Loop`, `IO_DISPLAY`, or `BYTES_PER_ROW` when those names make the idea clearer. The CPU reads the binary file, so the binary must contain only numeric bytes in the right order. The assembler stands between those two audiences. Its job is complete when every helpful name has been replaced by the right number and every source instruction has become the right machine byte.

This is the point where the language becomes more than a way to describe bytes. It becomes a practical tool for changing programs. A label can be moved by adding instructions before it. A constant can be defined once and used many times. A binary asset such as a font can be included without rewriting it as hundreds of numeric bytes. The machine still runs the same small instruction set, but the person writing programs no longer has to do every small piece of address arithmetic by hand.
